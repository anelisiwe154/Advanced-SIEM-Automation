# Template Analysis

## Comparison of GitHub Project Templates

| Template              | Columns (Default Workflow)               | Automation Features                          | Suitability for Agile |
|-----------------------|------------------------------------------|----------------------------------------------|-----------------------|
| Kanban                | Backlog, To Do, In Progress, Done                 | Auto-add sub-issues to project               | Simple visualization; good for Agile teams practicing continuous delivery and WIP limits |
| Team Planning         | ToDo, In progress, Done        | Auto-add sub-issues to project               | Structured sprint planning; better for larger teams managing backlog and upcoming cycles |
| Iterative Development | Backlog, Ready, In progress, In review | Auto-add sub-issues to project               | Strong fit for Agile sprints and iterative delivery; supports backlog and iteration planning |

---

## Justification

The Kanban template is best suited for this project. By offering a straightforward, lightweight depiction of tasks throughout stages (To Do -> In Progress -> Done), it conforms to Agile principles. Kanban encourages continuous delivery and places a strong emphasis on minimizing work-in-process, which helps avoid bottlenecks and guarantees constant development. Kanban provides the ideal mix of simplicity and Agile appropriateness because the SIEM automation system need flexibility and transparency rather than extensive sprint planning. It enables us to efficiently monitor workloads and user stories while swiftly responding to modifications.


# Reflection

## Challenges in Selecting and Customizing the Template
Selecting the template that best suited the team's workflow and project size was one of the challenges. Although Iterative Development and Team Planning provided further structure, they were a burden for a small team. Kanban was more straightforward, but in order to guarantee quality and visibility, columns like Review and Testing had to be added. Another difficulty was striking a balance between flexibility and clarity; too few columns can conceal crucial work stages, while too many can overwhelm a small team. In the end, customisation made it possible to adapt the board to Agile requirements.

## Comparison to Other Tools
Tools like Trello provide comparable Kanban boards to GitHub's templates, but they have various advantages:
With drag-and-drop cards and straightforward customization, Trello is incredibly visual and user-friendly. It does not, however, have a comprehensive integration with code repositories. During my final year, I tried this program, although it doesn't have as many deeper capabilities as GitHub projects. GitHub Projects is perfect for developers because it connects directly with problems and pull requests. Despite having fewer sophisticated reporting options than Jira, its ease of use and direct connection to code make it effective for small Agile teams.


## Conclusion
Choosing and modifying the Kanban template brought flexibility to light. Although GitHub's templates are more straightforward than Jira or Trello, small development teams using Agile can benefit greatly from their connection with problems and pull requests.
